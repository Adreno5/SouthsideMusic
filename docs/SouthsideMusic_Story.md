## 第一章 · 缘起

2025年10月，一个叫 Yanaka 的人开始了一个叫做 Southside Legacy 的项目——一个基于 Minecraft 1.12.2 和 Forge 的客户端。后来有个人加入了那个仓库，参与开发，也正是在那个仓库里，一个念头慢慢发芽。

这个念头跟 Minecraft 没有直接关系。它来自于另一个方向——音乐播放器。

在此之前，这个人——也就是我——已经写过一次播放器了。Python，PySide6，Pygame，用 requests 调网易云音乐的公开 API Demo。那个播放器跑得起来，但很丑，很粗糙，丑到自己都不好意思打开第二次。它躺在硬盘某个角落，像一个没写完的作业，每次看到都提醒你：你还欠自己一个像样的东西。

而另一边，网易云音乐的官方客户端一天比一天臃肿。启动慢，内存吃得越来越多，广告见缝插针。它本来只是一个听歌的工具，但不知道从什么时候开始，它变得不再只是一个工具了。

2026年2月12日——寒假还没结束，我打开 VSCode，新建了一个项目。

项目名字？Southside Music——南方音乐。名字是从 Southside Legacy 那里借来的，但内核是自己的：一个纯粹的、不臃肿的、属于我自己的音乐播放器。不是什么惊天动地的想法，只是想写一个让自己用着舒服的东西，仅此而已。

## 第二章 · 最初的模样

## 第三章 · 歌词、随机与图标

同一天下午，版本号跳到了 v3。

这个版本加了三样东西：歌词解析、随机选歌、还有图标文件。

歌词解析在当时是个硬骨头。本地音乐文件里的歌词嵌在各种格式里——LRC 是最常见的，但仍然有编码问题、时间轴偏移、多语言混排。我写了一个解析器，把那些 `[xx:xx.xx]` 格式的时间戳一行行啃下来，同步到播放进度上。当第一句歌词随着音乐亮起来的时候，那种感觉——自己写的东西在正确的时间做了正确的事——很难被别的什么东西替代。

随机选歌加了进来。不是技术上多难的事情，但它让这个播放器从"按顺序播完拉倒"变成了"可以一直听下去的东西"。一个 shuffle 按钮，带来的体验变化比想象中大得多。

图标文件是第一批 SVG。disc、playa、pause、next、last、music——每个按钮终于有了自己的脸。软件开始看起来像一个"软件"了，而不只是一个黑乎乎的调试窗口。

## 第五章 · 断舍离：告别 Pygame

2026年2月13日，我干了一件事：把 Pygame 从项目里整个挖掉了。

这不是一个轻松的决定。Pygame 是我最早学会的音频库之一，从第一个粗陋播放器到南方音乐的初期版本，它一直充当着声音引擎。但问题正在于此——它充当的是"引擎"，而不是"工具"。

最让我无法忍受的是 `set_pos` 和 `get_pos`。这两个函数是播放器里最核心的东西——设置播放位置和获取当前位置。但在 Pygame 里，它们像是在两个平行世界运行的。`set_pos` 接收浮点数，比如 `15.0` 表示跳到第 15 秒，耳朵听到的也确实跳到了 15 秒——但 `get_pos` 返回的是整数（时间乘以 1000 后的毫秒数）。假设音乐播到了 10 秒，`get_pos` 返回 `10000`。调用 `set_pos(15.0)` 之后，声音确实到了 15 秒，可 `get_pos` 还是返回 `10000`。

这两个函数只在音乐刚开始的时候校对一次，之后就各走各的路了。

这不是 bug，这是 Pygame 音频模块的设计上限。它的天花板太低，而我需要更精细的控制。

替换方案是用 pydub 做音频解码，用 sounddevice 做底层音频输出，然后自己手写一个 AudioPlayer。pydub 把音频文件转换成原始采样数据，sounddevice 负责推送到声卡，中间的所有逻辑——播放、暂停、跳转、变速、增益——都由我自己控制。

这是一次彻底的底层替换。播放器的心脏换了，但界面几乎没有变化。用户看到的是同一个播放按钮，但按钮背后已经不是同一个东西了。

提交信息里只写了一行："Replace pygame with custom audio player; optimize playback"。

## 第六章 · 预加载与波形之眼

替换了音频引擎之后，功能开始加速落地。

同一天，我加入了音频预加载。这是一个感知上的小改进，但体验差距很大——在 Pygame 时代，切换歌曲时会有明显的卡顿，因为解码发生在主线程，UI 会短暂冻结。预加载的思路是：在当前歌曲播放时，提前把下一首歌解码好放在内存里。切换时直接从缓存拿，而不是重新用ffmpeg加载。

缓存池限制在十首歌，用 LRU（最近最少使用）策略管理。这十首歌的缓存窗口，让切歌变得丝滑。代价是内存占用上升了一些，但在一个音乐播放器里，这是值得的。

同一天，FFT 频谱分析也上线了。

FFT——快速傅里叶变换。它将音频信号的时域数据转换到频域，告诉你每个频率上的能量分布。简单说，就是那个在播放器界面上随着音乐跳动的柱状图。

我用 `scipy.fft.rfft` 做变换，用 `numpy.hanning` 窗口减少频谱泄漏。每次音频回调时取一小块采样数据，送入 FFT 计算，然后把频率和振幅数据发射出去。界面端接收到这些数据后，实时绘制成频谱条。

这是南方音乐的第一个可视化功能。音乐不再只是耳朵能感知到的东西，眼睛也能看到了。

## 第七章 · 响度均衡——我最自豪的一笔

这是Yanaka喊着要要的功能。

LUFS（Loudness Units relative to Full Scale）是 ITU-R BS.1770 标准定义的响度计量单位。不同歌曲的录音响度参差不齐——有的歌声音大得炸耳，有的歌轻声细语。在歌单里随机播放时，这种落差让人很不舒服。响度均衡的目标是让所有歌曲听起来"音量一致"，而不是简单地把增益拉高拉低。

实现这个功能的关键是一个叫做 `pyloudnorm` 的库。我是通过 AI 了解到它的存在的。这个库提供了符合 EBU R128 标准的 Loudness Meter。但我没法直接把它作为依赖加进项目——它会引入一串神秘的依赖问题，编译时反复报错，排查到最后也无解。

于是我做了个折中：把 `pyloudnorm` 里的 `Meter` 类直接复制到项目里，就这样摆着。这个类实现了一整套 K-weighting 滤波器链、分块响度计算、双门控算法——从预处理滤波到绝对门限和相对门限的二级筛选，我全部忠实复制了 ITU-R BS.1770-4 规范。

每次播放歌曲前，先通过 Meter 测量整首歌的综合响度，然后计算出一个增益系数，使得目标响度（默认 -16 LUFS）与实际响度之间的差值被补偿回来。这个增益系数还被 `lru_cache` 缓存起来，同一首歌不需要重复计算。

效果是立竿见影的。随机播放一个包含古典、摇滚、流行、电子的大混合歌单时，再也不会出现"上一首歌震天响、下一首歌蚊子叫"的情况了。每首歌的听感响度被拉到了同一个水平线上。

在所有功能里，这是我最满意的一个。

版本号也走到了 v7。

## 第九章 · 数据的桥梁：WebSocket

二月十四日，南方音乐接入了一个新维度：WebSocket。

事情始于一个想法：如果播放器正在播放一首歌，能不能让另一个程序知道当前在播什么、歌词是什么、频谱长什么样？

WebSocket 是实现这个想法的桥梁。我在项目里嵌入了一个基于 Tornado 的 WebSocket 服务端。播放器启动时，这个服务也在后台启动，监听本地端口。任何连接到这个端口的客户端都能实时收到播放状态。

第一批通过 WebSocket 传输的数据是歌词。歌词同步需要毫秒级的精度——歌曲进度每向前推进一秒，当前的歌词行就要更新一次。通过 WebSocket 推送，比轮询优雅得多。

然后 FFT 频谱数据也上了 WebSocket。这意味着你可以在手机浏览器里打开一个页面，看到与电脑屏幕上完全同步的频谱跳动。音频数据在本地解码、本地处理、本地播放——频谱数据通过 WebSocket 广播到局域网内的任何客户端。

这个设计有点超前。当时我并没有想好谁会用到这个功能——但它就在那里，像一个打开的窗口。

## 第十章 · 从歌词到静音感知

v9 的标签是"Add lyric info type; optimize lyric update logic"。歌词系统在这个版本里得到了细化——歌词信息被抽象成独立的数据类型，更新逻辑被重写。之前的歌词同步基本上是"当前时间落在哪一行就高亮哪一行"，简单粗暴但够用。现在有了更精细的状态管理：歌词行有元数据，有空行标记，有版本信息。

与此同时，WebSocket 连接的管理也被重新设计。我写了一个 `SouthsideClient` 类来管理连接状态——连接、断开、重连、心跳。这些看似基础的网络基建，在之前是不存在的。当时的 WebSocket 连接是"连上就好，断掉拉倒"，没有任何状态管理。

v10 带来了一个别致的特性：静音跳过。

在真实使用中，很多歌曲的结尾有长达数秒甚至十几秒的空白——渐弱后的寂静、轨道间的间隙、现场录音的间歇。在歌单模式下，这些空白意味着你在等一首实际上已经结束的歌播完。静音跳过的思路是：当检测到音频音量持续低于某个阈值时，自动跳过当前歌曲。

实现方式是在音频回调中计算 RMS（均方根）电平，当电平低于阈值且歌曲已接近尾声时，触发 `onEndingNoSound` 信号，播放管理器收到信号后自动切到下一首。

这个功能做得很低调，但感受很强烈——歌单播放变得流畅了，空白被剪掉了，歌曲首尾相接，像一张混音带。

## 第十一章 · 阈值与标题栏

2月12日，v11。

v12 交付了这个功能。自定义标题栏意味着完全接管窗口的标题、边框、最小化/最大化/关闭按钮的绘制和交互。这在 PySide6 里需要移除原生窗口框架，然后用纯 Qt Widget 模拟窗口行为——包括拖拽移动、双击最大化、右键菜单等等。工程量不小，但效果也很直接：窗口的统一感提升了。

## 第十二章 · Island、FFmpeg 与 LFS

v13 阶段，一个奇怪的 bug 出现了。

Island——一个类似灵动岛的效果组件——它的复选框状态异常了。不是勾不上，也不是勾上后没反应，而是在某种特定操作序列下，复选框的状态和实际功能状态不同步。你看到它勾着，但实际上功能没开；你取消勾选，功能反而开了。

排查过程像在迷宫里找路。信号槽的连接顺序、状态初始化的时机、UI 和数据模型的同步节奏——每一条线索都指向不同的方向。最终定位到的问题和初始化顺序有关：组件的 `setChecked` 在被连接到信号槽之前就调用了，导致第一次状态变更没有被传播出去。

修完之后，我提交了一个信息简洁的 commit："Fix mysterious bug causing island checkbox state anomaly"。

同一批改动里，我把 FFmpeg 加入了项目仓库。

FFmpeg 是音频解码的核心依赖。之前用户需要自己安装 FFmpeg 并配置环境变量，这对非技术用户来说是一道门槛。把 FFmpeg 的二进制文件直接放进项目仓库，可以让播放器开箱即用，不再依赖外部环境。

但 FFmpeg 的二进制文件很大——接近 100 MB。这超出了普通 Git 仓库的承受范围。我启用了 Git LFS（Large File Storage）来追踪 FFmpeg 目录。`.gitattributes` 文件中添加了对应的规则。

在这一切之外，我还加了一个全局异常钩子。Python 程序崩溃时的默认行为是打印 traceback 然后退出。但在图形界面程序里，traceback 打印在控制台里，用户看不到。异常钩子把未捕获的异常显示在一个窗口上，这样即使程序崩溃，我也可以知道崩溃之前发生了什么。

## 第十三章 · 一首歌的名片

v14 发布于3月3日。

这个版本的标签是："Add send song info and cover image functionality"。

它做的事情说起来很简单：通过 WebSocket 发送当前歌曲的完整信息——标题、艺术家、专辑、封面图片。任何连接到 WebSocket 的客户端都可以收到这些数据。

封面图片是一个有趣的工程问题。歌曲文件的封面通常是嵌在文件本身里的——ID3 标签的 APIC 字段、FLAC 的 METADATA_BLOCK_PICTURE。mutagen 库可以提取出这些二进制图片数据，但需要转成 base64 编码后才能通过 WebSocket 的 JSON 消息传输。接收端再把 base64 解码成图片渲染出来。

v14 发布之后，南方音乐的提交历史出现了一个长达十一天的空白。三月三日到四月六日之间，没有一行代码被提交。

## 第十五章 · 启动、构建与暗色

v15 的内容集中在三个方向：启动体验、构建脚本、暗色模式。

启动窗口是全新的。之前的播放器启动后直接弹出主窗口——但如果加载的资源很多（封面、歌词、缓存），会有一段尴尬的空白期。启动窗口在这个空白期展示一个进度界面，让用户知道"程序正在加载，不是卡死了"。

构建脚本是另一个痛点。南方音乐需要用 Nuitka 打包成可执行文件分发。Nuitka 不是一个简单的打包工具——它把 Python 代码编译成 C，然后再编译成可执行文件。编译时间长，配置参数多，而且 Windows 上的路径处理、资源文件复制、依赖收集都很容易出错。

我写了一个 `build.bat`，把整个打包流程脚本化。同时调整了项目依赖配置，把不必要的依赖排除在打包之外，减少生成体积。

暗色模式自适应来自 `darkdetect` 库——它能检测操作系统的主题设置。当 Windows 切换到深色主题时，南方音乐也会自动切换。但这不止是一个开关切换——涉及到图标的深浅色适配、文字的对比度、背景的色值调整。深色主题不是简单地"把背景涂黑"，它需要考虑 UI 的层次感、可读性和一致性。

提交记录里有一条很简短："Fix LaunchWindow size and style"。启动窗口的大小和样式调了好几轮——太大显得笨重，太小信息显示不全，位置要居中。这些细节打磨花的时间往往比写功能本身还长。

## 第十六章 · 网易云音乐

2026年4月10日，南方音乐迎来了它诞生以来最大的一次转折。

v16。

之前的南方音乐只是一个本地播放器——你播放硬盘里已有的音乐文件，它负责解码、渲染、同步歌词。但本地音乐库总有放不满的时候，而网易云音乐（NetEase CloudMusic）拥有庞大的在线曲库。

我决定把网易云音乐接进来。

这不是第一次尝试——最早的、那个粗糙的旧播放器就用 requests 调过网易云的公开 API Demo。但那只是浅尝辄止。这次是认真的。

我找到了 `pyncm`——一个第三方的网易云音乐 API Python 客户端。它封装了网易云的登录、搜索、歌单、评论等接口。我把整个 `pyncm` 库作为 vendored 依赖塞进了项目，放在了 `src/pyncm/` 目录下。

这不是一个优雅的做法。vendored 依赖意味着你需要手动维护它的更新，而且它会让你的代码库膨胀。但当时我没有更好的选择——在一个平静的晚上我在开发时发现使用uv run无法运行main.py，一顿分析发现就在刚刚！作者把pyncm从PyPI和Github删除了，这对我是一个晴天霹雳，我立马翻找它的存档，最后找到一个副本。复制进来，改了就能用，这是最后的方案。

接入网易云 API 之后，南方音乐发生了质的变化。搜索框不再只能搜本地文件——你可以搜网易云的整个曲库。搜索结果以列表形式呈现，点一下就能搜索、缓存、播放。

我同时加了一个"会话页面"——这个页面管理着你和网易云之间的所有交互：登录态、搜索历史、播放记录。它像一个网关，连接着本地播放器和云端曲库。

从这一天开始，南方音乐不再只是一个本地播放器了。它是一个混合体——本地和云端、离线和在线的边界开始模糊。

## 第十七章 · 工具链的洗礼

网易云接入之后，构建和依赖管理的问题变得更加尖锐。

`pyncm` 引入了一整套加密和网络依赖——pycryptodome、requests 以及它们的传递依赖。打包体积暴增，Nuitka 编译时间翻倍。更麻烦的是，有些依赖在 Windows 上的预编译 wheel 缺失，Nuitka 在编译它们时会遇到各种奇怪的 C 链接错误。

4月11日，提交了一条记录："Update build scripts; optimize startup flow"——是为了加快构建速度而做的优化。

4月18日，更大的变动来了。我决定用 Nuitka 彻底重建整个构建流程，舍弃之前零散的打包配置，写一个统一的构建管线。Nuitka 会把 Python 代码编译成 C/C++，再编译成本地可执行文件。这意味着最终用户的电脑上不需要安装 Python、不需要安装依赖——单文件运行。

同一天，我移除了 `pyncm` 依赖。

不是放弃网易云功能——而是把 `pyncm` 的核心代码直接 fork 进项目，做成内置模块。这样就摆脱了对外部包管理器的不稳定依赖。代价是代码库又大了一些，但换来的是构建时的确定性和可复现性。

提交信息很简单："Remove pyncm dependency; optimize exception handling"。但背后是一次完整的构建工具链迁移。

## 第十八章 · 登录、VIP 与许可证

4月19日，v17。

这个版本带来了两个功能，都和网易云集成相关。

首先是网易云账号登录。网易云音乐的 API 有明确的权限分层：未登录用户可以搜索、试听低音质歌曲；登录用户可以获得高音质流；VIP 用户则可以获取无损音质（FLAC）。登录流程是通过二维码完成的——客户端生成二维码，用户用网易云手机 App 扫描确认，然后客户端轮询登录状态，获取认证令牌。

我还加了一个"权限标签"展示——在界面上显示当前账号的身份：普通用户、VIP、还是未登录。这个标签虽然小，但它让用户（也就是我自己）清楚地知道当前处于什么权限等级，为什么有的歌能播高音质而有的不能。

然后四月二十三日，另一条提交："commit。"。句号结尾，没有更多的话。有些提交就是这样——你说不清自己改了什么，或者改得太杂懒得分类，就丢一个占位符上去。

## 第二十章 · 自动与导出

5月1日，v19。

劳动节这天，我的小长假！南方音乐获得了自主生存的能力：自动下载缺失资源和自动更新。

先说服自己为什么需要"自动下载缺失资源"。随着网易云集成的加深，播放器需要在运行时下载音乐文件。但如果下载过程中断、网络不稳定、或者缓存被清理，播放器应该能自动重新获取，而不是弹出一个错误对话框让用户手动处理。我写了一个资源下载器，内置重试机制和进度跟踪。

歌曲导出功能也在同一天加入。本地缓存的网易云歌曲可以被导出到用户指定的目录，同时携带完整的元数据——标题、艺术家、专辑、封面、歌词。这是一个"取回自己数据"的功能：你通过播放器听到的歌，可以真正地下载到你的硬盘上，成为你的文件。

右键菜单随之而来。歌曲卡片上右键可以触发一系列操作：播放、添加到收藏、导出、查看详情。交互方式从"点击按钮"扩展到了"右键上下文菜单"，这是软件交互深度的提升。

v19 的标签是"Add auto-update feature"。自动更新功能诞生了。

自动更新的机制很简单：启动时检查 GitHub Releases 页面是否有新版本。如果有，后台下载新版本的可执行文件，下载完成后提示用户重启。这意味着即使我不在家里的电脑前，用户（包括 Yanaka 和未来可能的其他人）也能自动获得最新的版本。

同一天，提交里有一条信息："Fix file write position error"。一个关于文件写入位置偏移的 bug——在下载网络歌曲时，多线程写入导致数据块被写到错误的位置。修复很简短，但这个 bug 如果不修，导出的歌曲会出现音频断裂。

## 第二十一章 · 逐字与桌面

5月2日。高产的一天。

v20 带来了动态背景和元数据增强。播放器的背景不再是一张静态的纯色或图片，而是可以随着当前播放歌曲的封面色调变化——提取封面的主色调，生成主题色背景。同时 HTTP 请求配置被集中管理，不再散落在各个模块中。

交互式歌词查看器上线。之前的歌词是"显示在界面上，跟着进度走"。现在它是可交互的——你可以点击任意一行歌词跳转到对应的时间点，可以拖拽滚动浏览整首歌词。歌词查看器从被动显示变成了主动交互。

UI 动画也在这一天被大量优化。歌词滚动更加平滑，页面切换有了过渡效果。但用户立刻发现一个问题——播放页面的布局动画在频繁切换歌曲时会抖动。v20 的标签记录了这个修复："Disable playback page layout animation"。有时最好的优化就是关掉它。

v21 的重点是逐字歌词——YRC 格式。

YRC 是网易云音乐使用的逐字歌词格式。它比传统的 LRC 更精细：LRC 只能表示"这一行歌词从什么时候开始到什么时候结束"，而 YRC 能表示"每一个字在什么时候唱出来，持续多长时间"。逐字歌词的解析难度比 LRC 高一个量级——时间戳的粒度从行级降到了字级，虽然 YRC 的格式本身没有公开文档，但是我凭借我聪慧的头脑轻松看出它的token格式，开始时间在哪、什么格式，持续时间在哪等等。

逐字歌词的显示效果是：歌词亮起时，每个字依次高亮，像大厂级别的播放器一样。视觉上很细腻，但实现上需要精确的时间控制和逐字的位置计算。

v22 继续打磨这个功能——修复了逐字歌词的解析 bug 和当前行元数据颜色显示错误。

同一天，桌面歌词横空出世。

桌面歌词是独立于主窗口的、始终置顶的歌词浮窗。不管你在电脑上做什么——写代码、浏览网页、打游戏——歌词永远悬浮在屏幕的某个角落，同步滚动。

这个功能的来源可以追溯到第三章提到的 Island 组件。Island 是一个类似灵动岛的效果组件，它曾经出现过，但因为状态管理的 bug 被搁置了。桌面歌词是 Island 理念的继承者：它不再试图做一个花哨的动画效果，而是专注于一件事——让歌词始终可见。

实现上，桌面歌词是一个独立的无边框窗口，背景半透明，支持拖拽移动和透明度调节。它通过事件总线接收歌词更新，与主播放器保持同步。

5月3日发布 v23，标签是"updated to v23"。然后我把版本改成v21测试自动更新，但是我手滑提交了上去，所以立马 revert 了这次回退。

## 第二十二章 · 更新器、图标、许可证与事件总线

5月3日到4日，两天内版本号从 v23 跑到了 v26。

自动更新器经历了一次重大重构。之前的基础实现能工作，但在边界条件下不稳定——网络中断时的重试逻辑不够健壮、下载中断后没有断点续传、新版文件的完整性没有校验。我重写了启动器脚本 `launcher.py`，加入了更新校验机制：下载完成后计算 SHA-256 哈希，与服务器端提供的哈希值比对，只有完全匹配的更新才会被应用。

这次重构还产生了一个独立标签 `updator_hot_fix`——一个针对更新器的热修复。它专门解决更新逻辑中的一个竞态条件：当更新下载和用户手动启动几乎同时发生时，两个进程会争抢同一个文件的写入权限。

图标系统也在这个阶段被彻底重构。项目早期积累的 SVG 图标散落在各个目录中，命名混乱（出现了 `drop_down.svg` 和 `dropdown.svg` 同时存在的混乱局面），而且不同主题下需要不同的颜色。我写了一个统一的图标管理模块，所有的图标通过这个模块加载，而不是在界面上硬编码路径。

播放速度设置被加入到设置页面。之前变速播放只在代码中有开关，现在用户可以通过滑块在 0.1x 到 3.0x 之间调节速度。这个滑块的阻尼参数也经过了好几轮微调——太灵敏会晃动，太迟钝感觉不到反馈。

代码结构也被大幅重构。`main.py` 原本是一个千行级别的文件，塞满了各种初始化逻辑。我把它拆成了更小的模块——启动流程、组件注册、信号连接——各司其职。

5月4日，许可证从 MIT 切换到了 PolyForm Noncommercial License 1.0.0。

原因和二房东有关。南方音乐通过网易云 API 获取音乐内容，如果被第三方拿去做商业分发，责任可能会追到我头上。MIT 许可证允许任何人做任何事，包括商用。PolyForm Noncommercial 禁止商业用途，为项目增加了一层法律保护。这是在版权风险评估之后做出的决定。

同一天，项目的故事文档被创建——也就是你现在正在读的这个文件。空白的 `SouthsideMusic_Story.md` 文件被提交到仓库，等待被填满。

事件总线——`EventBus`——也在这段时间上线。这是一个简单的发布-订阅模式实现：组件之间不再直接调用对方的方法，而是通过事件总线的 `emit` 和 `on` 方法通信。解耦之后，播放器、歌词视图、桌面歌词、频谱显示之间的依赖关系变得更加清晰。任何一个组件都可以独立地被替换或修改，而不影响其他组件。

v26 发布了。两天之内从 v23 到 v26，节奏回到了寒假时的密集状态——五一假期真的太好了。

## 第二十三章 · 播放管理器与云歌单

v27 到 v29，5月4日到5月17日。这是南方音乐截至当前的最后一段旅程。

v27 调整了歌词滚动平滑因子——就是那个决定歌词从一行过渡到下一行时，动画是快是慢、是硬切还是缓动的参数。去掉了阈值检查，让平滑在任何速度下都生效。

同一天，断点续播的逻辑被重构。之前的实现是"记住最后一首歌"，重新打开软件时自动恢复播放。重构后增加了更完整的播放状态持久化——当前歌单、播放位置、播放模式（顺序/随机/循环）全部被保存和恢复。

v28 引入了一个更优雅的架构改动：用事件总线替换了分散的日志监听和主题切换逻辑。

在事件总线出现之前，主题切换是这样工作的：当用户在设置页面切换主题时，设置页面直接调用多个组件（播放控制器、歌词视图、频谱面板）的 `setTheme` 方法。这种硬编码的调用链让代码耦合度很高，每新增一个需要响应主题切换的组件，都要修改设置页面的代码。

事件总线解决了这个问题。主题切换时，设置页面只需要 `event_bus.emit(THEME_CHANGED, new_theme)`。任何关心这个事件的组件——不管是一个还是十个——都会收到通知。新增组件也不需要修改设置页面的代码，只需要让新组件订阅 `THEME_CHANGED` 事件即可。

同样被替换的还有日志监听。之前某些组件通过直接访问 Logger 实例来获取日志数据，现在改为通过事件总线广播日志消息。

v29 是当前最新的版本，发布在5月16日，也是改动最大的一次。

Playing Manager——播放管理器——是这次重构的核心。之前播放逻辑散落在 `PlayingPage` 和 `AudioPlayer` 之间。`PlayingPage` 既管界面又管播放控制，职责模糊。

Playing Manager 把播放控制的所有逻辑收归一处：歌曲切换、预加载、下载调度、播放模式、播放队列管理。`PlayingPage` 被瘦身，只保留界面渲染和用户交互的逻辑。`AudioPlayer` 回到纯粹的角色——只负责音频解码和输出。

云歌单功能是这个重构的直接产物。之前网易云集成只能搜索单曲播放，现在可以获取用户的整个云歌单列表——包括自己创建的、收藏的、以及网易云推荐的热门歌单。

我封装了一个 `NeteaseCloudMusicBackend` 类，提供统一的接口：获取用户歌单、创建歌单、编辑歌单、删除歌单、获取歌单内的歌曲列表。每一个操作都对应着网易云 API 的一个端点，对上层代码透明。

在 UI 上，歌单以卡片形式展示。每个歌单卡展示封面、名称、歌曲数量。云歌单卡片和本地文件夹卡片共享同一个父类，但在细节上有所区分——云歌单有"同步到本地"按钮，本地文件夹有"重命名"和"删除"按钮。

云歌单的歌曲不需要手动下载。点击播放时，播放管理器检查本地缓存——如果有，直接播放本地文件；如果没有，自动从网易云下载，然后播放。这个过程对用户几乎是透明的，唯一能感知到的差异是第一次播放时有一个短暂的加载延迟。
