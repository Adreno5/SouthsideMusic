# 南方音乐编年史

> Southside Music 的完整故事——从一个初中生寒假里的无聊尝试，到拥有自己音频引擎和加密协议的 35 个版本。

---

## 引言

2026 年初的 Windows 桌面音乐世界，说不上匮乏，也说不上美好。官方网易云客户端越做越重，启动越来越慢，内存越吃越多，广告位越来越密集。第三方替代品并不少，但大多出身凋零——作者热情褪去后便不再维护，留下一地断壁残垣。

但这篇编年史的起点，与这些宏大的行业叙事毫无关系。

它的起点，是一个七年级学生在寒假里感到无聊。打开电脑，打开 IDE，敲下一行 `import`——就像他之前无数次做过的那样。

从小学五六年级开始，他就断断续续地尝试写一些调用网易云 API 的小玩意儿。每次都是同一个配方：Python，同一个界面框架。每次也都是同一个结局：写了几百行代码，遇到一个过不去的坎，搁置，遗忘，文件夹里又多了一个永远不会再打开的目录。那些废弃的项目像一串省略号，散落在硬盘的角落。

他也说不清这次为什么不一样了。

也许是心态变了——之前是"玩玩看"，这次是"想做出来"。也许是 AI 的出现填平了那些曾经让他卡住很久的沟壑。也许只是因为他长大了几岁，手指能驾驭的复杂度变多了。总之，2026 年 2 月 12 日凌晨，他创建了一个 Git 仓库，把它叫做 Southside Music——一个随口起的名字，没什么特殊含义，就像那些随手打开又随手关掉的文件夹一样。

但这一次，他没有关掉它。

从冬天的寒假写到春天的开学，从开学写到暑假，从暑假继续往下写。4 个半月，35 个大版本，300 多个小时的深夜键盘声。他换掉了最初那个简陋的音频引擎，重写了网络通信协议，亲手实现了网易云官方的加密握手流程；他为歌词写了逐字对齐的解析器，为频谱画出了实时跳动的柱条，为每一首歌做了响度上的公平校正；他在某个深夜修复了一个纠缠了数月之久的崩溃 bug 后，在提交消息里写下了"I can't believe I fixed it"——用的是中文，语气是劫后余生。

这本编年史讲述的就是这个过程。不是最终产品有多么了不起，而是在那些看不见的时间里，一个人是怎么把一个"试试看"的念头，敲成了一万多行代码。

## 第一章 音频引擎：当游戏工具不再够用

Southside Music 的第一个音频引擎，选择几乎是不假思索的——pygame。不是因为 pygame 适合做音乐播放器（它不适合），只是因为这是当时唯一会用的东西。

那是一个能响就行的阶段。播放、暂停、下一首，三个按钮涵盖了全部能力。代码写在主文件里，和界面逻辑混在一起，像一张临时拼凑的桌子——晃晃悠悠，但勉强能放东西。当时他甚至调用的是别人搭建的第三方 API 中转站，整个播放链路是：应用 → 第三方中转 → 网易云服务器 → 返回音频 → pygame 播放。每一个环节都不在自己手里。

这种状态持续了大约一天。是的，只有一天。

v6 版本，上线不到 24 小时，pygame 被扫地出门。替换它的是一个基于 sounddevice 的自定义播放器，搭配 FFmpeg 做解码器。这个决定意味着什么？意味着他不再满足于"能响就行"，开始追求对音频管线的完全控制。也意味着他踏进了一片完全陌生的领域——音频采样率、缓冲区、声道映射、实时调度，每一个概念都要从头学起。

但代价来得很快。

替换音频引擎后不久，一个幽灵般的崩溃开始出现。播放一首歌，切到下一首，程序突然卡死，然后悄无声息地消失——没有错误弹窗，没有日志输出，没有任何线索。就像有人拔掉了电源。他在代码里加了一堆打印语句，逐个排查，却始终找不到规律。有时候连续播十首都没事，有时候播第二首就崩了。崩溃像一只随机出没的猫，无从预测，无从捕捉。

这个 bug 从 v6 开始，一直纠缠到 v20 左右。横跨了十多个大版本，持续了近三个月。

问题的根源后来才真相大白：sounddevice 的音频流在被销毁之前，不能加载新的音频数据。切歌时，旧流的销毁和新流的加载之间有一场微妙的竞速，一旦顺序出错，程序就会像踩到地雷一样直接爆炸——连"啊"一声都来不及喊。知道原因之后，修复只有一行代码。但找到这一行代码的过程，耗费了数十个小时的排查和无数次"这不可能"的自我怀疑。

音频引擎的第二波重大改造来自一个外部推动：流式播放。

大约在 v34 之前，播放器采用边下边播的模式——音频数据从网络到达后，一边写入缓冲区一边播放。这个方案本身没有问题，问题在于实现它的代码太过脆弱。那部分代码来自 AI 生成，结构混乱，边界条件处理一塌糊涂。每当网络波动或者用户快速切歌，整个管道就会堵塞，声音卡住，程序再次陷入那种他最熟悉的沉默式崩溃。

v34 是一个决定性的夜晚。提交消息只有一行——「REFACTOR!!!!oh my god!!!!!!!!!!!!!!!!!!!!!」——但这行文字背后是 124 个文件的变动，四千多行增删。核心决策只有一个：不再流式播放。改为完整下载后再播放。这个决定同时解决了一连串问题：切歌不再需要担心流状态，LUFS 响度处理不再受限于部分数据，WSOLA 变速算法也不再因为数据不完整而产生古怪的音频伪影。

说到 WSOLA，那是音频引擎的第三次蜕变。它允许多快或慢地播放一首歌，而人声不会变成花栗鼠或树懒。这项技术来自信号处理领域的经典论文，但对于一个初中生来说，论文中的数学推导如同天书。最终这条路是由 AI 铺通的——他描述需求，AI 生成代码，他再测试和打磨。效果出乎意料地好。

到 v35，音频引擎已经走完了一条完整的进化弧线：从 pygame 的"能响"，到 sounddevice 的"自定义管线"，到完整下载的"稳定播放"，再到 WSOLA 和混响效果的"专业处理"。当初选 pygame 只是因为"只会这个"，而现在引擎里的每一行代码都是他自己的选择。

## 第二章 通信协议：从借船到造船

在 Southside Music 的早期，播放一首网易云歌曲的路径是这样的：先向一个名叫 api.netstart.cn 的地址发出请求，再由它转发给网易云服务器，拿到音频数据后传回来。这是一个公开的第三方中转服务，好用，省事，不需要操心任何加密协议。但这也是一个黑盒——你不知道它什么时候会关停，什么时候会限速，什么时候会在你用户最多的时候突然返回 502。

所以，换掉它只是时间问题。

替代方案是一个叫做 pyncm 的开源库。他从 GitHub 上找到它，读了一遍——认出了一些熟悉的东西。AES 加密、RSA 加密、特定的参数拼装规则。网易云音乐的所有 API 请求都包裹在一层自定义加密协议中，这层协议通常被称为 WeAPI 和 EAPI。第三方中转做的事情，本质上就是在这个加密层外面套了一个 HTTP 接口，让客户端不用自己处理加密就能调用。

pyncm 在加密层之下工作。它不依赖任何第三方服务器，直接和网易云的后端对话。这意味着一件事：通信的主动权回到了自己手里。

但是，他其实并不完全理解那些加密算法。他知道 AES 是什么——在 pycryptodome 库里见过，能在代码里认出它的模样——但具体的密钥派发流程、请求签名的拼装规则，如果脱离源码让他从头复述一遍，他说不清楚。pyncm 对他来说更像一个运作良好的黑盒，只不过这个黑盒放在他自己的项目文件夹里，而不是某个陌生的服务器上。

真正的转折发生在 v34。

那时候，pyncm 的代码结构用了一种"包装器"模式：每个 API 接口返回一个包含端点和加密数据的元组，再由一个统一的发送函数去执行。这种写法本身没什么问题，但 Visual Studio Code 的 Pylance 类型检查器对此很不满——函数签名对不上，泛型推断失败，满屏的黄色波浪线。对于一个每天面对代码的人来说，那些黄色波浪线就像墙上的裂缝，不会让房子倒塌，但每看一眼都浑身不舒服。

于是一天深夜，他说服自己：与其将就着看这些警告，不如一次性改掉。他把整个 pyncm 的结构拆开，按照自己的编码习惯重新组装——不再返回元组，不再隐藏端点，每个 API 调用就是直来直去的函数调用。类型检查通过了，黄色波浪线消失了，代码看起来终于像"自己的"了。

那个提交改动了一百多个文件，四千多行代码。提交消息打了满屏的感叹号。但那不是愤怒，是一种终于解脱了的感觉。

## 第三章 歌词系统：当文字学会跳舞

歌词在音乐播放器里，长久以来扮演着一个配角。它存在，但没有人真正在意它——直到你开始在意。

Southside Music 最早的歌词支持很简单：从网易云 API 拉取一段 LRC 格式的文本，解析出时间标签和对应的句子，然后在一小块区域里逐行滚动。这是市面上九成播放器的做法，谈不上好坏，只是大家都不觉得歌词还能玩出什么花来。

但这位开发者有点不一样的想法。

v20 版本，他加入了一个"交互式歌词查看器"。这个词很绕口，但实际效果很直接：你可以在歌词区域滚动鼠标滚轮，自由浏览任意位置的歌词，点击某一行，播放器就会跳到那个时间点。这个功能本身不算新奇，但它意味着歌词不再只是一个"正在唱什么的显示器"，而变成了一个可以交互、可以导航的控件。

v21 紧接着带来了一个更微妙的变化：滚动不再生硬了。在此之前，歌词从一行切换到下一行的过渡是跳变的——上一行消失，下一行出现，简单粗暴。但 v21 引入了加速度平滑算法。歌词滚动时不再是匀速，而是根据剩余距离和时间动态计算速度：开始时快一些，接近目标时慢下来，最后轻轻"停靠"在新的行上。效果类似于物理世界中的惯性滑动。这个算法是他自己写的，不是从哪个开源库里扒来的现成方案。

真正的突破出现在 v22。

那一次，他盯着网易云歌词数据的原始格式看了很久。API 返回的歌词有两套格式：一套是传统的 LRC（每行对应一个时间点），另一套是 YRC——一种包含了每个字精确时间戳的格式。他"稍微动了动脑子分析 YRC 的格式"，就明白了它的结构。YRC 文件里，每一行歌词的每个字都携带了自己的开始时间和持续时间。这意味着，你可以在唱到某个字的瞬间，只高亮那个字，而不是一整行。

于是，逐字歌词诞生了。

效果是惊人的。播放一首歌时，歌词面板里的文字不是一行一行地变亮，而是像水波纹一样一个字一个字地漫过去。"你"亮起来，"好"亮起来，"吗"亮起来——每一个字踩在音乐的拍子上，精确到毫秒。这种体验在主流音乐播放器中都极为罕见，更别说在一个初中生写的软件里。

但逐字歌词的代价不低。YRC 格式的解析比 LRC 复杂得多，需要处理不规则的编码、缺失的时间戳、以及其他播放器导出的非标准数据。v22 之后连续好几个提交都在修复逐字歌词的各种边界情况——当前行的颜色不对了、解析器遇到某个符号崩溃了、某些歌曲的 YRC 数据结构不一样了。每一次修复都不大，但加起来，构成了歌词系统最密集的补丁期。

v29 又做了一次重要补充：歌词翻译对齐。网易云的 API 本身提供了翻译数据，但他需要把翻译行和原文行一一对应起来。如果原文有三行但翻译只有两行，或者顺序对不上，就需要额外的匹配逻辑。这项修正做好后，双语对照终于能够整齐地并排显示了。

他还有一个更长远的想法：对于那些 API 没有返回翻译的英文歌曲，未来可以嵌入一个 AI 模型，请求它实时生成翻译。这个计划还没有动手，但他已经把它写在了未来的蓝图上——不止是翻译，还包括用 AI 整理收藏夹、分类歌单。歌词系统不会是 AI 内嵌的唯一用户，但很可能是它最先落地的场景。

## 第四章 桌面歌词：飘在空中的一行字

主窗口里的歌词做得再好，也有一件事做不到：当你把播放器最小化去做别的事情时，你就看不到歌词了。

桌面歌词就是为了解决这个问题而生。

v23 版本第一次加入了这项能力。它本质上是一个新的窗口——一个脱离了主程序、永远浮在屏幕上方的独立窗口。这个窗口没有边框，背景不透明，只显示当前正在唱的那一句歌词。你可以用鼠标拖动它到屏幕的任何位置，如果拖到屏幕正上方，它会轻轻吸附在那里。

从表面看，它只是一个飘在桌面上的歌词条。但从代码层面拆开来看，它涉及了太多需要同步的事情。

最大的难关是流畅性。桌面歌词不是静态的文字牌——它需要和主播放器里的歌词保持完全同步，毫秒级的偏差都会让用户感到"歌词和声音没对上"。而且这个窗口是一个独立的进程内窗口，它通过事件总线接收歌词更新通知，这意味着一层额外的通信开销。如果通信有延迟，桌面歌词就会比主窗口的歌词慢半拍——这在视觉上非常扎眼。

v27 版本集中打磨了这方面的问题。桌面歌词加入了背景颜色混合效果，让文字在浅色和深色桌面背景下都能清晰可读。同时修复了滚动平滑度的绝对值计算错误，解决了在某些边界条件下歌词跳变的异常行为。还处理了屏幕刷新率变化时的适配——有些显示器是 60Hz，有些是 120Hz，桌面歌词的更新节奏需要跟着变。

到 v29，平滑滚动的动画又做了一次细化。之前修复过的问题没有再复发。桌面歌词从一个"刚做出来能用了"的功能，慢慢变成了"用起来还挺舒服"的功能。

它永远是主播放器的一个附件，而不是主角。但它的存在回答了一个简单的问题：当你把播放器最小化之后，窗口里那行正在唱的字，应该去哪儿。

## 第五章 响度均衡：让每一首歌站在同一起跑线上

听歌列表里如果同时有古典、摇滚、流行和民谣，一个让人烦躁的事情就会发生：每切一次歌，音量都会突然跳变。古典乐的轻声细语让你把音量拧大，下一首摇滚乐炸出来，差点把扬声器震破。这不是你的错，是不同录音的响度本身就不一样。

响度均衡要解决的就是这个问题。

这项技术的标准名称是 EBU R128，广播行业广泛使用的一个响度测量标准。它测量的是人耳感知到的"有多响"，而不是单纯的波形峰值。基于这个测量值，可以对音频做增益调整，让所有歌曲的感知响度趋于一致。

v8 版本，Southside Music 第一次引入了这个能力。实现的来源是一个叫做 pyloudnorm 的开源库。他没有直接把它作为依赖引入——那样太省事了，但也意味着项目多了一个甩不掉的外部包袱。他选择了另一种方式：把 pyloudnorm 的核心代码复制到自己的项目中，然后用 ruff 格式化工具重新整理了代码风格，让它的写法和项目里其他文件的写法一模一样。

这种方法有一个很实际的好处：他可以在复制之后随意修改。v10 版本，他调整了 LUFS 的应用逻辑——在重新应用新的增益值之前，先停止播放器，等处理完成再恢复。这个改动虽然小，但解决了一个实际的问题：如果一边播放一边调整响度参数，偶尔会出现音频毛刺。

真正让响度均衡变得完整的，是 v34 的架构变更。在流式播放时代，响度处理只能在下载到的部分音频数据上进行——如果一首歌只下载了前 30%，响度分析的精度就会受限。切换到完整下载模式后，整首歌的数据都在本地，响度测量可以覆盖整首曲目，增益调整的准确性提高了不止一个级别。

现在，不管列表里是轻柔的民谣还是轰鸣的摇滚，切换时的音量感受几乎是一致的。你不再需要频繁伸手去够音量旋钮。

## 第六章 频谱可视化：让声音现出原形

大多数人听歌时不会去想声音是什么形状的。但你一旦看到了，就会忍不住一直看。

频谱可视化是 Southside Music 底部常驻栏里的那道彩色动画——随着音乐的起伏，柱条像心跳一样跳动、扩散、收缩。它是全软件里"最不实用但最吸引人"的功能。

它在 v6 版本中第一次出现，和自定义音频播放器同时登场。这不是巧合：当你自己写音频引擎时，最原始的数据——音频采样——就在你手边。要把它变成频谱图，只需要经过一道傅里叶变换。这道变换将时间域的信号（声波随时间的变化）转换到频率域（哪些频率在哪个时刻有多强）。低频在左边，高频在右边，强度用柱条的高度表示。一切都遵循基本的数学原理。

v10 版本通过 WebSocket 把 FFT 数据实时传输了出去——这意味着外部的设备或程序也可以收到同样的频谱数据。同时加入了一个"FFT 倍数"配置项，让用户可以自己调整频谱的灵敏度。

v11 优化了频谱的绘制性能。在早期的实现中，每次动画帧更新时都要重新计算整个频谱的布局，这在 CPU 较弱的机器上会导致掉帧。优化后，计算被分摊到了更细的时间片上，动画不再和主界面的其他操作抢资源。

到 v23，频谱渲染被放到了异步线程中执行——和歌词加载共用同一个异步机制。这意味着频谱的更新不再阻塞主界面的其他操作，即使音乐在后台播放，软件的其他部分依然响应迅速。

它不会改善音质，不会让你听到更多细节，也不会帮你找到更好的歌。但它用一个直观的方式回答了"声音长什么样"这个问题——而那个答案，是动态的、彩色的、和你正在听的每一首歌一一对应的。

## 第七章 色彩：从专辑封面上偷一抹颜色

大多数音乐播放器的界面配色是固定的。开发者选一种主色，用户要么接受，要么换主题。Southside Music 走了一条不同的路：它不预设颜色，而是从正在播放的那首歌的专辑封面里"偷"颜色。

流程是这样的：当一首新歌开始播放，软件会下载它的封面图片，然后计算整张图片的像素平均值——把所有红色通道加起来除以像素数，绿色和蓝色同理——得到一个代表"这张封面整体是什么色调"的三元组。这个色调随后被应用到界面的各个元素上：播放面板的背景、进度条的颜色、高亮文字的颜色。歌换了，颜色也跟着换。

这个功能的核心代码只有不到四十行，依赖一个简单的数学工具：NumPy 的多维数组均值计算。但它带来的效果是持续且微妙的。每首歌都有了自己的色彩氛围，播放列表不再是同样灰度的条目排列，而是一段段不同颜色的记忆。

颜色的另一个维度是明暗。

软件通过 darkdetect 这个小小的库监听 Windows 系统的主题设置——如果系统切换到了深色模式，Southside Music 也会同步切换。这个切换不仅涉及背景色和文字色，还涉及整套图标系统。图标 SVG 文件本身只用黑色绘制，加载时根据当前主题把黑色替换为白色（深色模式）或保留黑色（浅色模式）。这样只需要维护一套图标源文件，就能自动适配两种模式。

v28 做了一个重要的改进：之前图标路径是硬编码的，深色模式和浅色模式各有一套图标文件。重构后，路径是动态生成的——根据当前主题实时计算 SVG 中的颜色替换，然后缓存到临时目录。图标不再需要在项目中维护两份副本，主题切换时的响应也更快了。

这套色彩系统没有使用什么前沿的技术。它的核心技巧是像素均值和字符串替换，都是再基础不过的操作。但它的设计思路是反常规的：不是让用户迁就软件的配色方案，而是让软件去迁就每一首歌。

## 第八章 搜索：在千万首歌里找到那一首

在接入网易云的完整曲库之前，搜索并不是一个有意义的功能。你只能搜自己下载到本地的歌，而本地的歌不需要搜索——你知道它们放在哪。

但 v16 改变了这一切。随着网易云 API 的接入，曲库从"几十首本地文件"膨胀到了"数千万首在线曲目"。搜索变成了入口，是用户和整个网易云曲库之间的第一道门。

搜索功能的实现，在代码层面很简单：输入关键词，调用 API 的搜索接口，拿到结果列表，展示在界面上。网易云的搜索接口本身质量很高——拼写纠错、联想补全、热门排序——不需要再做额外处理。Southside Music 需要做的就是把这些结果显示清楚：歌名、歌手、专辑封面。每一项都在一个卡片上呈现，点击纸飞机图标直接播放，点击爱心收藏到本地。

v29 为搜索框加上了一段展开动画——点击输入框时，它会从紧凑状态平滑展开为完整宽度。这个动画很小，但它改变了搜索的"手感"：搜索不再是一个冷冰冰的输入动作，而是一次有过渡感的界面响应。同时，搜索结果中的歌词匹配也做了改进——如果搜索关键词命中了一首歌的歌词翻译，那条翻译也会被高亮显示。

v33 把搜索的范围从"歌曲"扩展到了"歌单"。你现在可以搜索网易云上的公开歌单，找到后直接加载到左侧栏中。这个功能的代码量不大，但它改变了一个使用场景：之前你必须先登录、在自己的歌单里找歌；现在你可以像逛集市一样，搜索任何人的歌单。

搜索功能的演进是一个典型的"先有基础设施，再有上层功能"的过程。在 API 接入之前，搜索无处附着；API 接入之后，搜索自然生长。它不是被"设计"出来的，而是被"需要"出来的。

## 第九章 收藏夹：从文件列表到私人仓库

收藏夹是整个项目里年龄最大的功能之一。它和 Southside Music 同时出生，在 v1 的那个凌晨就已经存在了。

最初的收藏逻辑很简单：一个 JSON 文件，里面存着歌曲的名字和路径。加载时读这个文件，渲染成一个列表。保存时把当前列表写回去。它和一个文本文档没太大区别。

但正是这个简单的东西，经历了一轮又一轮的改造——每一次改造都不是因为收藏夹本身出了问题，而是因为软件的其它部分在成长。收藏夹作为所有歌曲的"家"，必须跟上这些变化。

第一次改造来自缓存架构的变化。早期的收藏夹把歌曲的音频数据和封面图片直接编码为 base64 字符串塞进 JSON 文件里。这种方法的好处是"一切都在一个文件里"，但缺点是 JSON 文件会随着收藏增多膨胀到几十兆字节，加载和保存都越来越慢。后来，音频和图片被移到了单独的文件目录中，JSON 只保存歌曲的元信息——名字、歌手、来源 ID。收藏夹的 JSON 立刻瘦了下来，从几十兆缩回到几十 KB。

第二次改造是结构重构。v26 对存储机制做了一次彻底的重写：歌曲的加载和缓存清理逻辑被抽出来重新组织，旧的"cache/"目录被迁移到新的"data/"目录体系下。为了不让老用户丢失已有的收藏，代码里保留了一段迁移逻辑——如果检测到旧的缓存目录存在，会先把里面的文件搬移到新位置，然后删除旧的目录。这就像搬家时先把旧箱子搬进新房子，再把空箱子扔掉。

第三次改造是事件化。v31 给收藏夹装上了"通知系统"：每当有歌曲被添加或移除，收藏夹会通过事件总线广播一条消息。界面上的其他组件——播放列表、左侧栏、搜索卡片——收到通知后会自动刷新。之前，这些组件需要在各自的地方轮询检查收藏夹有没有变化，既浪费资源又不及时。事件化之后，变化一发生，所有相关的地方立刻知道。

收藏夹的演进过程恰好映射了整个项目的演进方向：从"能存就行"到"存得好、读得快、通知得及时"。它始终是一个 JSON 文件，但围绕它的那套机制已经换了好几轮。

## 第十章 云歌单：本地与云端之间的桥梁

收藏夹解决了"把你喜欢的歌留在本地"的问题。但它遗漏了一个重要的场景：你最喜欢的歌，可能并不在你的本地收藏里，而是躺在网易云某个歌单里——也许是你自己整理的，也许是某个陌生人花了三年建的。

v29 之前，加载云歌单需要绕很远的路：登录网易云网页端，找到歌单 ID，复制，粘贴到一个输入框里。这个过程既不优雅也不直观。

v29 改变了这一点。左侧栏的收藏夹区域被重新设计，现在它同时展示两样东西：你创建的本地文件夹，和网易云账号下的云端歌单。点击云歌单，歌曲列表直接加载到播放区。不需要复制粘贴，不需要切换窗口。本地和云端第一次坐在了同一张桌子上。

与此同时，一个叫做 PlayingManager 的新组件在后台悄悄上线了。它不是用户能直接看到的东西，但它的职责非常重要：协调播放列表的来源究竟是本地收藏夹还是云歌单，以及当你在两者之间切换时，播放状态如何平滑过渡。在它出现之前，每次切换播放源都可能触发各种奇怪的边界条件——比如歌还没播完就切走了，或者列表清除后播放器还在试图读取已不存在的歌曲。PlayingManager 把这些零散的逻辑集中到一个地方，统一处理。

v33 又向前迈了一步。现在不只能看自己的云歌单，还能把云端的歌单下载到本地收藏夹；反过来，本地的文件夹也可以上传到云端变成一个歌单。歌单不再有"你的"和"云的"之分——它们只是同一批歌曲的两种存在形式。

配套的功能是歌单搜索——你可以搜索网易云上的公开歌单，找到后直接加载。这个过程和其他人的收藏发生了关联，让软件从一个私人播放器扩展为一个小小的音乐发现工具。

v34 的模型改动把"歌手"从一列字符串改进为包含名称和 ID 的对象。这个调整对普通用户来说看不见摸不着，但它让收藏夹里保存的歌曲信息变得更加完整和可靠——当你以后想根据歌手来整理歌单时，不至于因为名字重名而搞混。

## 第十一章 歌曲导出：给音乐一张身份证

Southside Music 最初的设计里，歌曲是"锁"在软件里的。你可以听，可以收藏，可以按你的习惯播放，但你拿不走——至少不能轻易拿走。

v19 加上的导出功能打开了这把锁。

导出的作用很简单：把一首歌保存成一个独立的音频文件，拿到任何设备上都能播。但这件事的背后远比"复制一份音频数据"要复杂。一首歌不是一个单纯的音频流，它由好几层信息构成：封面图片、歌名、歌手、专辑、曲目编号。这些信息应该嵌入到导出的文件中，而不是被丢在软件里。

v20 做得更彻底了一些。导出时，mutagen 这个库被用来写入完整的元数据标签：ID3 标签用于 MP3 格式，Vorbis 注释用于 FLAC，MP4 标签用于 AAC。封面图片被嵌入文件本身，而不是作为独立的图片文件放在旁边。导出的文件放到任何播放器中——手机的默认播放器、车载系统、别人的电脑——都能正确显示歌名和封面。

从用户体验的角度看，导出功能的入口也经过了设计。它不是藏在某个深层菜单里，而是出现在几乎所有歌曲卡片的右键菜单中。搜索列表里的歌可以导出，收藏夹里的歌可以导出，云歌单里的歌也可以导出。无论你是在哪儿看到一首歌，旁边都有一条路通向"把它带走"。

这个功能的代码量不大，大概两百行左右，但它回答了一个很实际的问题：如果有一天你不再用这个软件了，你收藏的那些歌怎么办？它们不会被困在里面。

## 第十二章 构建流水线：从 Python 源码到安装程序

Python 脚本在开发者的机器上跑得再好，对普通用户来说也是一堆无法直接运行的文本文件。要让一个不懂编程的人用上 Southside Music，必须经历一道工序：把 Python 源码打包成一个独立的可执行程序，让用户双击就能打开。

这个工序从项目的第一天就存在了。

v1 版本里的 build.bat 只有几行命令：调用 Nuitka（一个 Python 到 C 的编译器），把主文件编译成 exe，附带一些必要的资源文件。那时候的构建产物只是一个简陋的可执行文件夹，没有安装程序，没有图标，甚至没有正确的文件关联。

构建系统的演化是由一个个"少了点什么"的瞬间推动的。

v8 发现打包出来的程序缺少 NumPy——这是因为 Nuitka 不会自动识别所有动态导入的库，需要手动声明插件支持。打上插件标志后，频谱分析功能在打包版本中终于不再是黑屏。

v13 添加了 FFmpeg 的支持。FFmpeg 不是一个 Python 库，而是一个独立的命令行工具，需要用专门的资源目录机制打包进去。构建脚本多了一行"复制 FFmpeg 文件夹"的指令。

v17 对构建流程做了一次彻底重写。Nuitka 的调用参数被重新组织，输出目录结构变得更清晰，而且加入了更完善的资源复制逻辑。这次重写的动机很直接：之前的构建脚本是边写边改的产物，连他自己都快看不懂那些参数的含义了。

v23 加入了一个新的中间层：launcher.py。这个脚本不是主程序本身，而是一个"启动管理器"。它的工作是检查版本更新、验证文件完整性、然后才启动真正的程序。launcher.py 被 Nuitka 编译为主 exe，而 main.py 作为内部模块被调用。这样做的结果是：启动和更新逻辑独立于主程序之外，即使主程序崩溃了，启动器依然可以执行更新恢复。

v31 是构建系统历史中最重要的一站。它集成了 Inno Setup——一个 Windows 安装程序制作工具。构建流水线变成了这样：Nuitka 编译 → 复制资源 → Inno Setup 打包成一个 `.exe` 安装文件。用户不再需要手动解压文件夹，不再需要自己创建快捷方式。运行安装程序，一路"下一步"，Southside Music 就出现在了开始菜单里。

v35 又做了一次简化：不再把文件复制到复杂的嵌套目录结构中，而是直接平铺到 raw 文件夹里。构建脚本的指令减少了，错误也减少了。

从几行批处理命令到一个拥有安装程序、启动管理器、自动更新的完整构建流水线，这个过程花了四个月。每一步都是因为遇到了某个"少了点什么"的时刻。

## 第十三章 自动更新：软件自己会长大

一个不更新的软件，最终会变成一个废弃的软件。但如何让用户从旧版本移动到新版本，是一个比看起来复杂得多的问题。

Southside Music 的自动更新功能经历过两次完全不同的设计。

第一次尝试在 v19。思路很直接：检查 GitHub 仓库的最新版本号，如果比当前版本新，就下载更新文件，覆盖本地的对应文件。这个方案的问题是它太脆弱了——如果更新过程中断（网络断开、程序关闭、磁盘空间不足），本地的文件可能处于一种"新旧混搭"的状态，程序再也无法正常启动。

v23 发现了一个在测试中漏掉的问题：更新机制没有妥善处理"版本号回退"的情况——比如从 v22 降级到 v21。此外，启动器脚本对更新文件的校验不够严格，损坏的下载文件也可能被当作有效更新安装。这两个问题被修复后，自动更新第一次达到了"可用"的状态。

但真正的改进发生在 v30。

那是一次架构级别的重写。之前的方案是逐个文件更新——新版本和旧版本对比，只下载有变化的文件。听起来聪明，但实现起来需要维护一份文件清单和校验和表，一旦清单本身出了错，整个更新机制就失灵了。而且，逐个文件更新的速度被大量小文件的网络延迟拖累，更新一个版本可能要下载上百个零碎文件。

v30 的方案简单粗暴得多：不再按文件更新，而是按发布包更新。每次新版本发布时，在 GitHub Releases 上传一个完整的压缩包。更新程序检测到新版本后，下载这个压缩包，解压，覆盖整个程序目录。一次下载，一次解压，一切结束。这个方法浪费了一点带宽（每次都下载完整的程序），但它彻底消除了"新旧文件混搭"的风险——要么成功更新到完整的新版本，要么停留在完整的旧版本，没有中间状态。

同期还做了一件事：更新程序自身的更新机制。launcher.py 被设计为可以自我替换——当检测到启动器本身有更新时，先下载新版本的启动器，标记为"待替换"，然后在下一次程序启动时执行替换。这让自动更新链条上的每一个环节都有了升级的能力。

v33 又清理了更新检查中的一些冗余逻辑——去掉了不必要的检查间隔配置和排他规则。更新系统变得更安静了，不会再在后台发出多余的请求。

到现在为止，自动更新已经默默工作了好几个版本。用户打开软件，如果后台有新版，它会在下次启动时悄然完成替换。大多数用户甚至注意不到这个过程——这正是它设计的目标。

## 第十四章 WebSocket 服务：向南边的人发消息

大多数音乐播放器只管自己放歌。但你如果想把"现在在播什么"的数据发给另一个设备、另一个程序、另一个屏幕，就需要一条额外的通道。

这条通道就是 WebSocket 服务。

v10 版本，Southside Music 内部出现了一个基于 Tornado 框架的 WebSocket 服务器，监听在 15489 端口上。它不做什么复杂的操作，只做两件事：接收连接，发送数据。

发送的数据有两类。一类是歌词同步信息——当歌曲播放时，当前行歌词、当前时间戳、歌手和歌名会被打包成一条消息，推送给所有已连接的客户端。另一类是 FFT 频谱数据——实时计算的频率振幅数组，同样通过这条通道发送出去。连接的客户端可以在自己那一端重建同样的频谱动画。

这个功能的存在，源自一个很实际的场景：他可能想在另一个设备或另一个程序上展示 Southside Music 正在播放的内容。比如说，在手机上显示桌面歌词，或者在另一个窗口中画一幅更大更精细的频谱图。WebSocket 就像一根管道，把播放器的内部状态泵到外面去。

后来加入的连接状态管理功能让用户可以手动控制是否接受外部连接——需要时打开，不需要时关掉，避免不必要的网络活动。

这个服务的使用范围可能很窄——它服务的不是普通用户，而是那些想把数据拿到别处去用的人。但对于那些确实需要它的人来说，它是一条现成的、不需要额外搭建的通道。

## 第十五章 配置系统：记住你的一切

一个软件用得越久，它需要记住的东西就越多。窗口位置、音量大小、上次播放到哪首歌、播放模式是循环还是随机、响度目标值是多少。如果每次打开软件这些设置都要重新来一遍，它永远无法成为一个"常用"的工具。

Southside Music 的配置系统从 v4 开始独立存在。它的核心是一个 JSON 文件——config.json——放在项目的根目录下。软件启动时读它，运行时随时写它，关闭前保存最后一次。

这个 JSON 文件的内容随着版本的演进而不断膨胀。v4 时它只记录了几个基础选项；到 v35，它已经包含超过四十个字段。从语言选择到音频输出设备，从桌面歌词位置到频谱灵敏度因子，从登录会话到立体声增强开关——几乎每一个用户可调节的旋钮都在 config.json 里有一个对应的位置。

配置系统的设计有一个明显的特点：它偏爱平铺结构而非嵌套。所有字段都直接放在 JSON 的根对象下，而不是按功能分组到子对象中。这样做牺牲了一点组织性，但换来的是读取和写入时的极简代码——不需要递归遍历，不需要路径解析，一个 `cfg.__dict__.update(data)` 就能把整个配置加载进来。

有两个功能值得一提。

第一个是"续播"。config.json 里保存着上次播放的歌曲列表、当前播放的索引、以及这首歌已经播到了第几秒。软件启动时，如果检测到这些数据，会问你是否要继续上次的播放。这个功能在 v28 被重构为使用事件总线来恢复状态，保证无论是正常退出还是意外崩溃，播放位置都会被记住。

第二个是旧配置迁移。早期的配置使用 Python 的 pickle 格式存储（config.pkl），后来切换到 JSON。v26 的重构中加入了检测旧 pickle 文件的逻辑——如果发现 config.pkl 存在，会尝试读取并迁移内容到 config.json，然后删除 pickle 文件。这是面向"持续了多个版本的老用户"的考虑：他们不会因为配置格式变了而丢失所有设置。

v32 移除了一个不那么必要的功能——自动保存。之前 config.json 每秒检测一次是否有外部修改，如果有就自动重写。这个功能的初衷是让用户可以直接编辑 JSON 文件来修改配置，但实际使用中几乎没有人这样做。去掉之后，配置只会在程序关闭和设置变更时保存，减少了无意义的磁盘写入。

## 第十六章 事件总线：看不见的信号网络

在 v28 之前，Southside Music 的组件之间是这样通信的：页面 A 直接调用页面 B 的方法，页面 B 直接调用页面 C 的方法。这种"硬连接"在只有几个页面时没问题，但当页面数量超过十个，每个页面又包含子组件时，依赖关系就成了一团乱麻。你改一个播放按钮的行为，可能会有三个其他页面因此出问题。

v28 引入了一个叫做"事件总线"的机制来解开这团乱麻。

事件总线的原理很简单：任何组件都可以"订阅"一个事件，当别的组件"发布"这个事件时，所有订阅者都会收到通知。事件发布者不需要知道谁会收到消息，订阅者也不需要知道消息是谁发的。A 和 B 之间不再有直接的函数调用，取而代之的是一座看不见的信号塔。

这套机制实现起来只有不到八十行代码。核心是一个字典，把事件名称映射到订阅者的函数列表上。发布事件时遍历这个列表，依次调用。就这么简单。但在这八十行代码之上，运行着整个软件的跨组件通信。

到 v35，事件总线上注册了近六十种不同的事件类型。有关于播放的：歌曲变更、播放结束、下一首、上一首、加载完成、播放出错；有关于主题的：主题即将切换、主题已切换、语言变更；有关于收藏夹的：收藏变更、文件夹增删；有关于歌词的：歌词行变更、刷新率变化；还有关于调试的、关于加载状态的、关于 WebSocket 连接的。

事件总线还处理了一个 Qt 特有的麻烦：QObject 生命周期。Qt 的组件可能在任何时候被销毁，如果一个已经销毁的组件的函数仍然挂在事件总线的订阅列表上，程序就会崩溃。事件总线在每次发布事件时会自动检查订阅者是否"活着"——通过 shiboken6 库验证底层的 C++ 对象是否仍然有效——如果发现已销毁的订阅者，会自动将其从列表中移除。

一个特别的设计是事件总线可以整体禁用。这在某些情况下很有用：比如在批量修改配置时，不希望每一个字段的变更都触发一次界面刷新。关闭事件总线，做完所有修改，再打开。

从 v28 开始，新的功能几乎都是通过事件总线接入系统的。新页面不需要知道有哪些其他页面存在，只需要订阅它关心的事件。组件之间的"硬连接"变成了"软连接"——每一段连接都可以随时断开和重建，而不影响其他部分。

## 第十七章 登录：推开那扇门

网易云音乐的 API 有一个严格的规则：未登录状态下，几乎所有歌曲都只能试听 30 秒。这意味着，不登录的 Southside Music 几乎是一个空壳——你能搜到歌，但听不完。

登录功能就是为推开这扇门而生的。

v16 版本加入了会话页面，它提供了两种登录方式：手机号和二维码。手机号登录需要输入号码，获取验证码，填入验证码——和你在其他任何软件上的登录流程一样。二维码登录则是在屏幕上生成一个二维码，用网易云手机 App 扫描即可。

这两种方式的底层原理不同。二维码登录的流程稍显复杂：每隔几秒轮询一次服务器的登录状态，直到用户扫码确认或者超时。在等待期间，会话页面需要保持一个清晰的进度提示，让用户知道"正在等待扫码"——而不是像卡住了一样静止不动。

v17 在登录的基础上加上了 VIP 权限检查。登录后，软件会检查当前账号的会员等级，并根据权限决定是否允许播放某些高品质音频。这个功能在 v31 被移除了——不是因为它实现得不好，而是因为它的存在违背了这个项目的初衷：让听歌这件事变得简单。

v33 利用登录状态做了一个更实际的事情：音频缓存的重新验证。如果你是 VIP 会员，下载的音频可能是高品质版本；如果你的会员过期了，之前缓存的高品质音频就不能再直接使用。登录后根据当前身份重新检查缓存的有效性，避免播放"过期"的音频数据。

会话数据——包括登录令牌、登录方式、登录状态——被存储在 config.json 中，和其他的配置项放在一起。这样每次启动都不需要重新登录，除非令牌过期或者用户主动退出。

推开那扇门之后，三十秒的限制消失了。用户可以搜索、播放、收藏、导出——做的每一件事，都是站在登录这个基础之上的。

## 第十八章 启动窗口：第一次见面很重要

在 v16 之前，Southside Music 的启动方式很"程序员"：一个黑色的控制台窗口闪一下，然后主界面弹出来。没有过渡，没有提示，没有任何"我正在努力加载"的信号。如果启动过程中发生了某个缓慢的操作——比如网络请求超时——用户看到的就是一个空白窗口，不知道软件是卡住了还是在工作。

v16 加入的启动窗口改变了这一点。它不是一个加载动画，而是一个带有具体进度信息的界面：正在加载配置、正在检查更新、正在初始化音频、正在恢复上次的播放状态。——每一步都有一条文字提示。

启动窗口的实现藏着一个细节：它不只服务于用户，也服务于事件总线的初始化。事件总线在创建时可以接收一个指向启动窗口的引用。在启动过程中，每一个组件向事件总线订阅事件时，都会向启动窗口发送一条进度消息。这意味着启动窗口上显示的那些文字，是各个组件"自己报上来"的——而不是一个预制的脚本。真实的进度，真实的顺序。

v26 对启动体验做了两处改进。一是降低了进度更新的间隔——之前每完成一步要等一小段时间才更新文字，现在几乎瞬间响应。二是增加了启动日志的输出粒度——更多的细节被写入日志文件，如果启动出了问题，排查起来更容易。

启动窗口在 v35 的构建简化中也获得了受益。构建脚本不再生成嵌套的目录结构，启动器路径更加直接，减少了启动时文件查找的开销。

启动窗口存在的全部意义就是一件事：在软件准备好之前，给用户一个信号，告诉它一切都在按计划进行。它只存在几秒钟，然后功成身退。

## 第十九章 国际化：让世界听得懂

Southside Music 从第一天起就默认使用中文界面。这很自然——开发者是中国人，大多数用户也是中国人，为什么要考虑其他语言？

但 v35 版本的一个提交改变了这个默认前提。国际化支持——简称 i18n——在那时被加了进来。

i18n 的实现方式非常传统：一份翻译字典，把每个需要翻译的文本字符串映射到不同语言的版本。这份字典被定义在一个 Python 文件中，包含英文和简体中文两个版本。软件启动时根据 config.json 中的语言设置加载对应的翻译。

翻译字典的规模不小——英文和中文各有大约五百条条目，覆盖了从设置页面的选项说明到错误弹窗的提示文字，从右键菜单的操作名称到桌面歌词的控制面板。每一条条目都有一个唯一的键名，按照所属的页面和功能分组：`setting_page.playback_speed`、`favorites_page.add_to_folder`、`error_popup.details`。

这套方案的技术含量不高——没有 gettext，没有 ICU 消息格式，没有语言包热加载。但它覆盖了所有用户能看到的文字。语言切换在设置页面中即时生效，不需要重启程序。

字典里有一个有趣的细节：它不只包含了面向用户的文本，还包含了一些面向开发者和调试场景的文字。比如 `setting_page.larger_value_means_more_intense_changing`——这种条目显然是写给调试频谱可视化时自己看的，后来也被纳入了翻译体系。

到 v35 为止，国际化还是一个"框架就位、内容待补"的状态。英文翻译的质量尚需打磨，更多的语言还不在考虑范围内。但地基已经打好了——从今以后，每一句新的界面文字都会带着自己的翻译键名出生，而不是直接硬编码为中文。

## 第二十章 调试与错误处理：软件也需要一面镜子

软件和人一样，生病时最怕的是说不清哪里不舒服。用户说"它崩了"，开发者问"崩之前你做了什么"，用户说"什么都没做，它就崩了"——这种对话在软件世界里每天发生无数次。

Southside Music 的错误处理哲学是：别让用户帮你找 bug。

v13 版本加上了第一道防线：异常钩子。Python 没有捕获的异常会触发 `sys.excepthook`——Southside Music 接管了这个钩子，当程序遇到未处理的异常时，不再直接崩溃退出，而是弹出一个错误窗口，把异常的类型、消息、调用堆栈整整齐齐地列出来。窗口下方还有一个按钮，点击后直接跳转到 GitHub Issues 页面，异常信息已经预先填好了。用户只需要写一句"我刚才在做什么"，然后点击提交。

但这只是事后补救。更重要的，是在软件运行时就照清楚它内部的状态。

v34 加入的调试系统承担了这项任务。它的工作方式是通过事件总线向所有组件广播一条"收集调试信息"的消息，各个组件收到后，把自己当前的状态整理成文本发回来。频谱引擎报告当前的 FFT 倍数值，音频播放器报告当前输出设备的名称，WebSocket 服务报告连接的客户端数量。所有这些信息被汇总到一个调试窗口中，每隔 20 毫秒刷新一次。

这意味着，如果软件出了某个不易复现的问题——比如频谱画到一半不动了——可以打开调试窗口，看到频谱引擎此刻的各参数值，判断问题是出在 FFT 数据的来源上还是出在绘制通道上。不需要猜测，不需要反复尝试，数据就在那里。

这套调试系统的设计思路是把"异常"和"异常前兆"区分开来。异常钩子处理的是已经崩了的情况，调试系统处理的是"快要崩了但还没崩"的情况。捕捉"前兆"比收拾"后果"更有价值。

日志系统是第三道防线。每个模块都有自己的日志记录器，按照严重程度分级：错误、警告、信息、调试。大多数时候日志只记录错误和关键事件，但当需要排查深层问题时，可以调高日志级别，看到每一个函数调用的细节。日志文件保存在磁盘上，即使程序完全崩溃退出了，日志也还在，可以在下次启动时查看。

这三道防线——异常钩子、调试系统、日志——共同构成了一套从"将崩未崩"到"已经崩了"再到"崩后排查"的完整覆盖。它们确保了一件事：遇到问题时，开发者不会两眼一抹黑。

## 第二十一章 静音跳跃：不让沉默拖慢节奏

大多数歌曲的结尾都有一段安静的部分——和弦淡出、回声消散、或者纯粹是录音结束后留下的空白。在顺序播放或随机播放时，每首歌结尾的这段沉默会在切换前被完整体验一遍。如果你在一个晚上听五十首歌，每首歌结尾有三秒空白，那就是两分半钟的沉默。

v10 加入的"智能跳过无声片段"专门处理这件事。

它的工作原理很直接：在歌曲即将结束时——具体提前多少秒由用户设定——开始检测音频的音量。如果音量持续低于用户设定的阈值超过一定时间，播放器就会认为"这首歌已经结束了"，提前切换到下一首。阈值和提前检测时间都可以在设置页面中调整，适应不同风格的歌曲：古典乐的自然弱音段落不会被误判为"结束"，而流行乐末尾的长淡出可以毫不犹豫地跳过。

这个功能不是替换掉原有的歌曲结束检测，而是在它之上叠加了一层判断。正常播放时一切如常，只有当"剩余时间不多且音量很低"两个条件同时满足时，它才会介入。

用户可以选择开启或关闭这个功能。不想要的人可以关掉，继续听完每一首歌的最后一个音符；想要的人开着它，让列表的切换更加紧凑。同一个问题，两种答案。

## 第二十二章 声音的调味料：立体声、混响与音调

音频引擎把歌播出来，只是及格。真正让这个播放器与众不同的，是它允许你改变声音的"味道"——不是通过均衡器调节高低频，而是通过更特殊的处理。

v8 加入的立体声增强是第一个声音调味料。它的原理源于一种叫做 Haas 效应的听觉现象：如果左右声道接收到同一个声音的时间有极微小的差异——几毫秒到几十毫秒——人耳会感知到声场变宽了，声音不再像是从两个扬声器中间的某一点发出的，而是扩散到了更广阔的空间。实现的代码是 AI 生成的，描述里写着"Haas"这个词，配置里有一个以毫秒为单位的延迟指数。

v34 带来了更多调味料。混响效果模拟了声音在不同空间中的反射：在一个小房间里唱歌，声音是干的、紧的；在一个大礼堂里，同样的声音会带着嗡嗡的回声。混响强度可以从设置面板中调节，从"几乎听不出来"到"像在浴室里唱歌"。

同一次更新带来的还有多选播放列表——你可以一次性选中多首歌，然后统一添加到收藏夹或导出。这个功能在操作层面改变了使用习惯：以前是一首一首地点，现在可以批量处理。

v35 加入的音调调节是最后一个重要调味料。它和变速播放是两回事：变速改变播放速度的同时也会改变音调（声音变高或变低），而音调调节则是在不改变速度的前提下，只改变音调的高低。你可以把一首歌升调或降调几个半音，听起来像换了一个歌手在唱——但节奏完全不变。这个功能需要和 WSOLA 算法配合工作：WSOLA 负责在变速时保持原始音调，而音调调节反过来，在保持速度的前提下改变音调。

这三个功能——立体声增强、混响、音调调节——有一个共同点：它们都不是播放器的"必需品"。没有它们，Southside Music 依然是一个功能完整的音乐播放器。但有了它们，软件的回答从"我能播"变成了"你想怎么播"。这个转变，比任何单个功能都更有意义。

## 尾声：还没完，继续走

2026 年 2 月 12 日凌晨，一个七年级学生创建了一个 Git 仓库，随手取了个名字，打算写个"小东西玩玩"。他之前写过无数次类似的项目——打开 IDE，写下几行 import，遇到一个过不去的坎，然后关掉文件夹，再也没有打开过。

但这一次，他没有关掉。

四个半月后，这个"小东西"长到了三十五版、一万多行代码。它的音频引擎从 pygame 走到了 WSOLA 和音调调节；它的网络层从调用别人的中转 API 走到了自己实现加密通信；它的歌词系统从逐行滚动走到了逐字高亮和桌面悬浮窗；它的构建流水线从一个批处理脚本走到了带安装程序和自动更新的完整链路。它学会了自己检查错误、自己翻译界面、自己从专辑封面里偷颜色。它学会了很多开发者好几年都未必能接触到的东西。

但技术成长只是这个故事的一个维度。另一个维度是：一个人在面对一堆不知道答案的问题时，没有选择关掉窗口。

不知道音频流为什么崩溃——那就查了几个月，直到找到那一行代码。不知道网易云的加密协议怎么写——那就读别人的源码，认出 AES，搞定它。不知道 WSOLA 的数学推导——那就让 AI 搭桥，自己验证结果，再打磨到能用。不知道构建脚本为什么失败——那就重写一次，不行就再重写一次。

每一个"不知道"的后面，都跟着一个"但还是做出来了"。

开发者在回顾这段历程时说，最大的收获是技术能力的提升。但这话说得太谦虚了。技术能力提升是结果，不是原因。原因是他在寒假某个深夜决定，这个文件夹不会再被关掉。

项目还没有结束。他想给它嵌入 AI，用来翻译没有翻译的歌词、整理杂乱的收藏夹。安装程序已经有了，但还有很多细节可以打磨。这些计划写在了他的脑海里，没有明确的截止日期，也没有必须完成的压力。这不是一个即将交付的产品，这是一个还在生长的东西。

他说，对这软件想说的话只有一句：「还没完，继续走。」

这篇编年史写到这里，也该搁笔了。但 Southside Music 的日志文件还在继续写——下一次启动时，它又会多出一行新的记录。那行记录是什么，目前还没有人知道。不过有一个预测是安全的：它不会是最后一行。

真正的功夫，都在看不见的时间里。而看不见的时间里，还有更多的功夫要下。
